## 1. Общая картина

Telegram-бот — «цифровой двойник» владельца, эксперта по контенту и контент-стратегиям. Умеет:

- отвечать **текстом** в стиле и от лица владельца (мозг — Claude);
- отвечать **голосом** владельца (клон в ElevenLabs);
- понимать **входящие голосовые**;
- отправлять **видео-кружочки** с аватаром владельца (Sora 2 / Veo — переключаемые провайдеры): два режима — «лайфстайл» (владелец в локации: кафе, машина, улица) и «ответ» (часть ответа в формате видео);
- использовать **базу знаний** владельца (папка .md-файлов);
- вести **отдельный диалог (чат) с каждым пользователем**, помнить контекст.

### Стек

| Слой | Технология |
|---|---|
| Язык | Python 3.11+, asyncio |
| Telegram | aiogram 3.x (polling) |
| LLM | Anthropic API, `claude-sonnet-4-6` |
| TTS / STT | ElevenLabs |
| Видео | Провайдеры: Sora 2 (OpenAI), Veo (Google) — через общий интерфейс |
| Медиа | ffmpeg (subprocess) |
| БД (локально) | SQLite |

| Конфиг | python-dotenv |

## 2. ЭТАП 1 — Текстовый бот с персоной, базой знаний и разделением по чатам

**Цель: бот отвечает текстом как владелец, опираясь на базу знаний, и ведёт отдельный диалог с каждым пользователем.**

### 2.1. Мозг (brain.py)

- Anthropic Messages API, `claude-sonnet-4-6`, официальный SDK `anthropic`.
- Системный промпт = три блока:
  1. **Каркас роли** (в коде): «Ты — цифровой аватар владельца. Отвечаешь от первого лица, как живой человек в Telegram-переписке. Не упоминай, что ты ИИ, если не спрашивают прямо. Кратко и по делу: обычно 2–6 предложений, без markdown-заголовков и списков, разговорным языком. Вне своей экспертизы — честно скажи.»
  2. Содержимое `persona.md`.
  3. База знаний (всё из `knowledge/`, склеено с разделителями `=== ФАЙЛ: имя.md ===`).
- **Обязательно prompt caching** (`cache_control` на блоках persona+КБ) — они повторяются в каждом запросе. Сверь синтаксис с актуальной докой Anthropic.
- `max_tokens`: 1024. При ошибке API — 1 повтор, затем вежливое сообщение.

### 2.2. База знаний (persona_kb.py)

- При старте грузить все `.md`/`.txt` из `knowledge/` в память.
- Оценить объём (символы/3). Если **< ~80 000 токенов** — класть всё целиком в промпт (режим по умолчанию).
- Если больше — НЕ ломаться: режим tool use, где Claude получает список файлов с краткими описаниями (первые 2 строки) и инструментом `read_knowledge_file(filename)` подгружает нужные. Реализуй, активируй только при превышении.
- `/reload_kb` (админ) — перечитать папку без перезапуска.

### 2.3. Разделение по чатам и память (memory.py)

**Важно: каждый пользователь = отдельный диалог (чат) с собственной историей.**

- Ключ диалога — `chat_id` Telegram (для лички = id пользователя). Если позже понадобятся темы/треды внутри одного пользователя — заложи поле `thread_id` (по умолчанию `default`), но в Этапе 1 один тред на чат.
- Таблица `messages(chat_id, thread_id, role, content, created_at)`.
- В контекст Claude — последние `HISTORY_MESSAGES` (30) сообщений ИМЕННО этого чата. Истории разных пользователей строго изолированы.
- `/reset` — очистить историю своего чата.
- Таблица `users(chat_id, username, first_seen, last_seen)` — заводить при первом сообщении.

### 2.4. Обработчики (handlers.py)

- `/start` — короткое приветствие от лица владельца.
- Любой текст → ответ через brain.py с историей данного чата.
- Индикатор «печатает…» (`send_chat_action`) во время генерации.
- Антифлуд: не более 1 одновременной генерации на чат (остальное — «секунду, отвечаю на предыдущее»).
- Только личные сообщения. В группах вежливо отказываться.

### 2.5. Persona.md (создай шаблон)

Секции для заполнения владельцем: `# Кто я`, `# Моя экспертиза`, `# Стиль общения` (тон, длина, эмодзи, ты/вы), `# Фирменные фразы`, `# О чём не говорю`, `# Примеры ответов на типовые вопросы`.

### 2.6. Админ (admin.py) — уже в Этапе 1

Только для `ADMIN_TELEGRAM_ID`:
- Отправка боту документа `.md`/`.txt` → сохранить в `knowledge/` (имя = имя файла, перезапись) → авто-reload → ответ «База обновлена: файл X, всего Y, ~Z токенов».
- `/kb` — список файлов с размерами. `/kb_delete имя` — удалить.
- `/persona` — отправка файла заменяет `persona.md`.
- `/stats` — число пользователей, сообщений за сегодня, сгенерировано голоса/видео.

**Приёмка Этапа 1:** бот запускается; отвечает в стиле персоны; использует факты из тестового файла КБ; у двух разных пользователей независимые диалоги; `/reload_kb`, загрузка файла, `/reset` работают.

## 3. ЭТАП 2 — Голос

**Цель: бот понимает голосовые и отвечает голосом владельца.**

### 3.1. Исходящий (voice.py)
- ElevenLabs TTS, `ELEVENLABS_VOICE_ID`, актуальная multilingual-модель с русским (сверь со списком моделей).
- Конвертация в **OGG/Opus 48kHz mono** через ffmpeg (требование Telegram).
- `send_voice` + `send_chat_action("record_voice")`.
- Если ответ длиннее `VOICE_MAX_CHARS` — отправлять текстом. Ошибка ElevenLabs → текст с пометкой «(голос временно недоступен)».

### 3.2. Входящий
- Скачать voice → распознать через ElevenLabs Scribe. Если в актуальной доке Scribe недоступен — остановись и предложи владельцу альтернативу (OpenAI Whisper), пусть выберет.
- Распознанный текст обрабатывается как обычное сообщение.

### 3.3. Выбор формата ответа
- Claude сам выбирает формат: первой строкой ответа служебный тег `[FORMAT:text|voice|video]`, далее ответ. Тег вырезается перед отправкой.
- Правила: по умолчанию `text`; пользователь прислал голосовое или просил голос → `voice`; явная просьба формата — подчиняться. `video` — см. Этап 3 (до него `video` принудительно → `voice`).

**Приёмка Этапа 2:** голосовое на входе распознаётся; ответ голосом владельца; текст по-прежнему работает.

## 4. ЭТАП 3 — Видео-кружочки (Sora 2 + Veo, асинхронно)

**Цель: бот по запросу присылает кружочек с владельцем — в режиме «лайфстайл» (в локации) и «ответ» (часть ответа в видео).**

### 4.1. Интерфейс провайдера (video/base.py)

```python
class VideoProvider(Protocol):
    async def generate(
        self,
        script_text: str | None,   # что говорить (None = немой лайфстайл-клип)
        scene_prompt: str,         # локация/действие: "сидит в кафе, говорит в камеру"
        audio_path: str | None,    # аудио ElevenLabs для лип-синка (если есть)
    ) -> str:                      # путь к mp4
        ...
```

- Фабрика выбирает провайдера по `VIDEO_PROVIDER`. Оба провайдера реализованы, переключение — одной переменной.

### 4.2. Veo (video/veo.py) — вход: ФОТО

- Идентичность владельца — **набор фото из `identity/`** (НЕ одно фото: дрейф лица; используй несколько референсов / bind-subject, если API поддерживает).
- `scene_prompt` задаёт локацию и действие.
- Если нужен говорящий кружок с голосом владельца — генерим сцену, затем накладываем лип-синк под `audio_path` (video/lipsync.py). Если провайдер умеет нативно — используем нативно.

### 4.3. Sora 2 (video/sora2.py) — вход: CAMEO

- **Важно:** Sora НЕ принимает загруженные фото лица. Идентичность = `SORA_CAMEO_ID` (записанный владельцем cameo). Сверь по доке актуальный способ ссылаться на cameo в Video API.
- `scene_prompt` через @-упоминание персонажа/cameo. Сцена и действие — в промпте.

### 4.4. Сборка промпта сцены
- Claude (в brain.py) при `[FORMAT:video]` дополнительно отдаёт: короткий `script_text` (для говорящего кружка, ≤ `VIDEO_MAX_CHARS`) и `scene_prompt` (локация — из контекста или дефолтная «нейтральный домашний фон, смотрит в камеру»).
- Лайфстайл-режим: если пользователь просит «кружок как ты в кафе» без вопроса по сути — `script_text=None` или короткая фраза, `scene_prompt="кафе"`.

### 4.5. Асинхронная очередь (bot/jobs.py)
- Видео генерится 1–4 минуты → **никогда не блокируем диалог**.
- Поток: пользователь просит видео → бот СРАЗУ шлёт текст/голосовой ответ + сообщение «🎥 записываю кружочек, ~минуту…» → кладёт задачу в таблицу `video_jobs(id, chat_id, status, script, scene, provider, created_at)` со `status='queued'`.
- Воркер (asyncio-таск внутри того же процесса) берёт queued-задачи, вызывает провайдера, поллит статус, скачивает mp4, постобрабатывает, отправляет `send_video_note`, ставит `status='done'`. Таймаут 4 мин → `status='failed'` + фоллбек: отправить голосовой/текст вместо видео.
- Конкурентность воркера: 1–2 задачи одновременно (не больше, чтобы не ловить rate limit провайдера).

### 4.6. Постобработка (video/postprocess.py)
- ffmpeg: обрезка в квадрат, масштаб **640×640**, H.264+AAC, длительность ≤ 60 сек.
- `send_video_note`.

### 4.7. Экономия и лайфстайл-кэш
- `VIDEO_DAILY_LIMIT_PER_USER` (по умолч. 2) кружочков на пользователя в день; сверх — голос вместо видео.
- Папка `video_clips/`: заранее заготовленные лайфстайл-кружочки (приветствие, «доброе утро», заглушки). Бот сначала проверяет, нет ли готового подходящего клипа, и отдаёт его без генерации. Приветственный кружочек нового пользователя — из кэша.

**Приёмка Этапа 3:** «запиши кружок, что такое контент-воронка» → через ~минуту приходит кружочек с владельцем, ≤60 сек, нормального качества; переключение `VIDEO_PROVIDER=veo|sora2` работает; дневной лимит соблюдается; диалог не блокируется на время генерации.

## 5. ЭТАП 4 — Деплой (Render + Supabase)

**Профиль сервиса:** постоянно живущий процесс + фоновые видео-задачи + ffmpeg. Поэтому:

### 5.1. Render — дом бота
- Тип: **Background Worker** (HTTP не нужен, бот на polling). Один процесс: aiogram-поллинг + asyncio-воркер очереди.
- Бесплатный план засыпает при простое — для бота не годится; нужен платный always-on (Starter).
- ffmpeg: установить (apt в build-команде или базовый образ с ffmpeg).
- Все ключи — через Environment в дашборде Render, не в коде.

### 5.2. Supabase — данные и файлы
- **Postgres**: таблицы `users`, `messages`, `video_jobs`, `daily_counters`. Схема в `db/schema.sql`.
- **Storage**: bucket `identity` (фото-референсы), bucket `videos` (кэш сгенерированных).
- `db/store.py` — абстракция: локально SQLite (если `SUPABASE_URL` пуст), в проде Supabase. Один интерфейс, две реализации.

### 5.3. Vercel — НЕ для бота (на будущее)
- Serverless-таймауты и отсутствие постоянного процесса делают Vercel непригодным для самого бота. Зарезервировать Vercel под будущий лендинг/админ-дашборд «Ilfat Sistema». Тяжёлые видео-задачи в любом случае остаются на Render.

### 5.4. Локальная разработка (MacBook M1)
- Всё запускается локально: `python main.py`, `brew install ffmpeg`, БД — SQLite, файлы — локальные папки. Облако нужно только для постоянной работы. Дай инструкции запуска для обоих режимов (local / prod).

**Приёмка Этапа 4:** бот стабильно работает на Render 24/7; данные в Supabase; переживает перезапуск (история и задачи не теряются).

## 6. Чего НЕ делать
- Без векторных баз, эмбеддингов, RAG-фреймворков (объём КБ это не требует).
- Без Redis/Celery — очередь в Postgres-таблице.
- Без вебхуков — polling.
- Не хранить временные медиа дольше суток — чистить `data/tmp` при старте.
- Не отвечать в группах.
- Не привязывать всю видеологику к одному провайдеру — только через интерфейс `VideoProvider`.
