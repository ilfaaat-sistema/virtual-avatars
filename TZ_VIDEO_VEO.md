## 1. Что строим

Бот отправляет видео-кружочки (`sendVideoNote`) с аватаром владельца в двух режимах:

- **Лайфстайл (основной режим):** владелец в локации (кафе, машина, улица), НЕ говорит в камеру в упор. Видео генерит Veo (silent / no visible speaker), поверх — закадровый голос из ElevenLabs. Лип-синк не требуется → нет рассинхрона.
- **Говорящий ответ (вторичный режим):** короткая реплика в камеру (≤6 секунд речи). Veo генерит «чистый диалог» (двигаются губы), затем аудио Veo заменяется на дорожку ElevenLabs через ffmpeg. Синхрон «достаточный», не идеальный — это осознанный компромисс.

Идентичность владельца — фото-референсы из папки `identity/`, приоритетно мастер-картинка (см. 2.2). Передаём в Veo как reference images / ingredients.

**Требование владельца (зафиксировано):** в кружочке всегда должны быть И видео, И голос владельца (из ElevenLabs). Неидеальный лип-синк в talking-режиме принят осознанно — для MVP это приемлемо, голос важнее точного синхрона. Оба режима всегда озвучиваются голосом ElevenLabs, собственный звук Veo не используется никогда.

## 2. Архитектура

### 2.0. Сквозной поток обработки видео-запроса (читать первым)

Это полный путь от сообщения пользователя до прилетевшего кружочка. Все модули ниже (2.1–2.7) — части этого потока; собирать их строго в этом порядке.

1. **Пользователь пишет** (например «запиши кружок, как ты в кафе, и расскажи про контент-воронки»).
2. **Мозг (brain.py)** формирует ответ и ставит тег `[FORMAT:video]`. Дополнительно отдаёт служебные поля:
   - `mode` = `lifestyle` или `talking`;
   - `scene` = локация/действие («кафе у окна, тёплый свет»); если не указана — дефолт;
   - `spoken_line` = короткая реплика для озвучки (≤250 симв.), либо текст закадра.
3. **handlers.py** сразу, не дожидаясь видео, отвечает пользователю текстом или голосом по сути вопроса + отдельным сообщением «🎥 записываю кружочек, ~минуту…».
4. **jobs.py** кладёт задачу в таблицу `video_jobs` (Supabase) со `status=queued` и всеми полями (chat_id, mode, scene, spoken_line).
5. **Воркер (jobs.py)** берёт задачу → `processing`:
   a. **veo.py** берёт мастер-картинку из `identity/master.*` как reference image, собирает промпт из `scene` + жёстких требований композиции «под кружочек» (квадрат 1:1, лицо по центру, субъект не уходит из кадра) → генерит видео Veo 3.1 Lite (720p, без звука) → скачивает mp4.
   b. **voice.py / compose.py** генерит голос владельца через ElevenLabs по `spoken_line` → ffmpeg глушит звук Veo и накладывает дорожку ElevenLabs (лип-синк приблизительный — принято осознанно).
   c. **postprocess.py** умный кроп по лицу → 640×640 → mp4 под Telegram.
6. **Воркер отправляет** готовый кружок через `sendVideoNote` отдельным сообщением → `status=done`.
7. **При ошибке/таймауте** → `status=failed` → фоллбек: отправить ответ голосом/текстом + «не получилось записать кружок, отвечаю голосом».

Данные между шагами: `brain → (mode, scene, spoken_line) → video_jobs → veo (scene+identity→video) → compose (spoken_line→audio→mux) → postprocess (640×640) → sendVideoNote`.

### 2.1. Провайдерная абстракция (video/base.py)

```python
from typing import Protocol, Optional

class VideoProvider(Protocol):
    async def generate_scene(
        self,
        scene_prompt: str,            # локация/действие: "сидит в кафе у окна, тёплый свет, смотрит в сторону"
        identity_images: list[str],   # пути к фото из identity/
        talking: bool,                # True = режим говорящего (clean dialogue, видны губы), False = silent лайфстайл
        spoken_line: Optional[str],   # реплика (только для talking=True, ≤6 сек речи)
        duration_sec: int,            # целевая длительность (MVP: ~8 сек)
    ) -> str:                         # путь к сырому mp4 от провайдера
        ...
```

- Фабрика выбирает провайдера по `VIDEO_PROVIDER` (`veo` по умолчанию). Сейчас реализуем только `veo`, но интерфейс готов под будущие (D-ID/Kling) без переписывания.

### 2.2. Veo-провайдер (video/veo.py)

- **Модель: Veo 3.1 Lite** (самый дешёвый тариф, ~$0.03–0.05/сек) через Gemini API или Vertex AI. Точное имя модели уточни в актуальной доке (Veo 3 / Veo 2 deprecated и отключаются 30.06.2026 — использовать только линейку 3.1). Имя пишется в `VEO_MODEL`.
- **Генерация БЕЗ звука и в 720p** — обязательно. Звук Veo нам не нужен (всё равно глушим и кладём ElevenLabs), а отключение аудио и 720p заметно снижают цену за секунду. Если в доке есть флаг `audio: false` / параметр без аудио — использовать его. 4K не нужен.
- Передать `identity_images` как reference images / «Ingredients to Video» (до 3 — проверь лимит в доке).
- **Идентичность лица — приоритет входа (важно для снижения дрейфа):**
  1. **Предпочтительно — мастер-картинка персонажа (character sheet / master image):** одна чистая, консистентная по свету и стилю картинка владельца. Это даёт самый стабильный результат между клипами. Если в `identity/` лежит файл с именем `master.*` (или `character_sheet.*`) — использовать в первую очередь как основной reference.
  2. **Иначе — набор из 2–3 чистых фото** владельца (фронт + ракурсы), переданных как мульти-референс.
  3. **Никогда — одно случайное фото:** это даёт максимальный «морфинг» лица.
  - Реализуй логику выбора: сначала ищем `master.*` в `identity/`, при отсутствии — берём до 3 фото из папки. Залогируй, какой режим идентичности использован.
  - Дрейф лица всё равно полностью не убирается (Veo — не обученный аватар), но мастер-картинка заметно стабильнее набора случайных фото.
- **КОМПОЗИЦИЯ ПОД КРУЖОЧЕК (критично!).** Кружок — это квадрат 640×640, поэтому Veo должен генерить кадр СРАЗУ пригодный под кружок, а не случайное горизонтальное видео, из которого потом вырежется край плеча. В КАЖДЫЙ промпт (оба режима) обязательно добавлять:
  - соотношение сторон **1:1 (квадрат)** через параметр aspect ratio API (если 1:1 недоступно — 9:16 вертикаль, она кропается в квадрат с меньшими потерями, чем 16:9);
  - **крупный или средний план, лицо в центре кадра и в верхней половине**, голова целиком в кадре с небольшим запасом сверху, плечи видны;
  - формулировки вроде: «square 1:1 framing, centered close-up portrait, face fills upper-center of frame, head fully visible, looking toward camera».
  - **Действия должны быть совместимы с форматом говорящей головы:** даже если владелец ест, смотрит на дорогу за рулём или поднимает голову задумавшись — лицо ОСТАЁТСЯ в кадре и по центру, субъект НЕ уходит из кадра, НЕ отворачивается полностью, камера держит лицо. Действие — лёгкий фон к портрету, а не главный сюжет. В промпт добавлять: «subject stays centered and in frame at all times, camera keeps the face framed, natural subtle movement only».
- Сборка промпта:
  - **talking=False (лайфстайл):** в промпт добавить указания тишины/без речи («no visible speaker, no dialogue, ambient only»), описать локацию и естественное действие — НО с соблюдением требований композиции выше (лицо в кадре, по центру). Звук Veo нам не нужен — всё равно заменим/заглушим.
  - **talking=True:** реплику оформить по правилам Veo (прямая речь в кавычках с глаголом речи), добавить «clean dialogue only, no background music», крупный/средний план, чтобы губы были видны и лицо было по центру.
- Генерация асинхронная: submit → poll статуса → скачать mp4. Таймаут 4 минуты.
- Длительность: для MVP целимся в нативные ~8 сек (не усложнять scene extension). Если нужно длиннее — НЕ реализуй сейчас, оставь TODO.

### 2.3. Озвучка и сборка (video/compose.py)

> **КРИТИЧНО — про голос.** Свой/клонированный голос в Veo НЕ подаётся: у Veo нет входа для аудио-сэмпла, «Ingredients to Video» работает только с картинками, а не со звуком. Veo генерит собственный голос, который нам НЕ нужен. Поэтому голос владельца всегда добавляется ОТДЕЛЬНЫМ шагом постобработки: Veo даёт видео → его звук глушим/удаляем → накладываем дорожку ElevenLabs. Никаких попыток «передать голос внутрь Veo» — это невозможно, не трать на это время.

- Реплику/закадр генерит ElevenLabs (переиспользуй `bot/voice.py`, тот же `ELEVENLABS_VOICE_ID`). Получаем аудио-дорожку с голосом владельца.
- ffmpeg:
  - заглушить/удалить оригинальную дорожку Veo (она нам не нужна ни в каком режиме);
  - наложить дорожку ElevenLabs;
  - если длительность речи ≠ длительности видео — выровнять: подрезать/добавить тишину, чтобы аудио не выходило за видео (для лайфстайла это нормально; для talking стараться, чтобы речь укладывалась в клип).
- **Лип-синк в talking-режиме будет неидеальным** (губы синхронны под звук Veo, а голос — ElevenLabs). Это принято осознанно: владельцу важно, чтобы в кружочке были его видео И его голос, даже ценой неточного синхрона. Не пытайся «чинить» синхрон сложными средствами — короткие реплики (≤6 сек) делают рассинхрон приемлемым.

### 2.4. Постобработка для Telegram (video/postprocess.py)

- **Умный кроп по лицу, НЕ слепой центральный.** Veo уже отдаёт квадрат/вертикаль с лицом по центру (см. 2.2), но подстраховаться на выходе обязательно, иначе владелец может оказаться обрезанным:
  - определить положение лица в кадре (детекция лица, например OpenCV/Haar или mediapipe) и кадрировать квадрат вокруг лица — лицо в верхней-центральной части;
  - если детекция не сработала (лицо не найдено) — фоллбек: кроп по верхней-центральной части кадра, НЕ по геометрическому центру (лицо обычно выше центра);
  - масштабирование в режиме «cover» (заполнять кадр), без чёрных полей.
- Затем: масштаб **640×640**, H.264 + AAC, длительность ≤ 60 сек.
- Результат — mp4, готовый для `sendVideoNote`. Перед отправкой залогировать, что лицо найдено и в кадре (для отладки композиции).

### 2.5. Асинхронная очередь (bot/jobs.py)

Видео генерится 1–4 минуты — **диалог не блокируем**.

- Таблица в Supabase `video_jobs`:
  `id, chat_id, status (queued|processing|done|failed), mode (lifestyle|talking), scene_prompt, spoken_line, created_at, updated_at, error`
- Поток:
  1. Пользователь просит видео (или Claude выбрал `[FORMAT:video]`).
  2. Бот СРАЗУ отвечает текстом или голосом по сути + отдельным сообщением «🎥 записываю кружочек, ~минуту…».
  3. Кладёт задачу в `video_jobs` со `status=queued`.
  4. Воркер (asyncio-таск в том же процессе) берёт queued → `processing` → Veo → compose → postprocess → `sendVideoNote` → `done`.
  5. Таймаут/ошибка → `failed` + фоллбек: отправить ответ голосом/текстом и сообщить «не получилось записать кружок, отвечаю голосом».
- Конкурентность воркера: 1–2 задачи одновременно (избегаем rate limit Veo).
- Воркер должен переживать перезапуск процесса: при старте подхватывать незавершённые задачи (`processing` старше N минут → вернуть в `queued` или `failed`).

### 2.6. Выбор формата (правка bot/brain.py)

- Сейчас `[FORMAT:video]` принудительно превращается в `voice` (заглушка). **Снять заглушку.**
- Claude при `[FORMAT:video]` должен дополнительно вернуть служебные поля (например, второй строкой после тега):
  - `mode`: `lifestyle` или `talking`;
  - `scene`: краткий scene_prompt (локация/действие); если пользователь не указал — дефолт «нейтральный домашний фон, тёплый свет»;
  - для `talking` — текст реплики (≤ ~250 символов, чтобы уложиться в ≤6 сек речи).
- Правила выбора режима: если пользователь просит «как ты в кафе/машине/на улице» без вопроса по сути → `lifestyle`. Если просит видео-ответ на конкретный вопрос → `talking` с короткой репликой. По умолчанию при `video` — `lifestyle`.

### 2.7. Лимиты и кэш

- `VIDEO_DAILY_LIMIT_PER_USER` (по умолчанию 2). Считать в Supabase (таблица счётчиков или COUNT по `video_jobs` за сутки). Сверх лимита → вместо видео отвечать голосом + «лимит кружочков на сегодня исчерпан».
- Папка `video_clips/` — заранее заготовленные/реально снятые кружочки (приветствие, «доброе утро», заглушки). Перед генерацией бот проверяет, нет ли подходящего готового клипа, и отдаёт его без вызова Veo. Приветственный кружок нового пользователя — из кэша.

## 4. Риски (зафиксированы осознанно)

- **Дрейф лица.** Veo с фото-референсами не гарантирует идентичное лицо между клипами. Минимизируем мульти-референсом (3 фото) и короткими клипами. Принимаем как ограничение MVP.
- **Лип-синк в talking-режиме «достаточный», не идеальный** (губы под звук Veo, голос — ElevenLabs). Минимизируем короткими репликами ≤6 сек. Лайфстайл-режим этой проблемы не имеет.
- **Стоимость.** Veo 3.1 Lite без звука в 720p — порядка $0.03–0.05/сек, то есть ~$0.24–0.40 (≈18–30 ₽) за 8-секундный кружок. Дешёвый тариф выбран осознанно; для кружка 640×640 качества Lite достаточно. Дневной лимит + кэш `video_clips/` сдерживают расходы. Тестовый прогон — на $300 trial-кредитах Google Cloud (бесплатно).
- **Длительность.** Нативно ~8 сек; длиннее — только scene extension (НЕ в MVP).
- **Зависимость от Veo.** Вся видеологика — через `VideoProvider`, чтобы при необходимости добавить D-ID/Kling/лип-синк-слой без переписывания.

## 5. Критерии приёмки

1. «Запиши кружок, как ты сидишь в кафе и говоришь про контент-воронки» → через ~минуту приходит кружок: владелец в кафе, закадровый голос ElevenLabs, ≤60 сек, 640×640, диалог не блокировался.
2. Talking-режим: короткий видео-ответ в камеру с голосом владельца, синхрон приемлемый.
3. **Композиция кружочка:** во всех сгенерированных кружочках лицо владельца в кадре, по центру, не обрезано — даже когда он выполняет действие (ест, за рулём, поднимает голову). Субъект не уходит из кадра.
3. Дневной лимит соблюдается; сверх лимита — голосовой фоллбек.
4. Готовый клип из `video_clips/` отдаётся без генерации.
5. Этапы 1–2 (текст, голос) работают как раньше.
6. Незавершённые задачи переживают перезапуск воркера.

## 6. Чего НЕ делать

- Не реализовывать scene extension / длинные видео в MVP.
- Не добавлять сторонние лип-синк-сервисы сейчас (оставить как будущий провайдер).
- Не блокировать диалог на время генерации.
- Не привязывать логику к Veo напрямую — только через `VideoProvider`.
- Не хранить сырые/временные видео дольше суток — чистить временные файлы.
