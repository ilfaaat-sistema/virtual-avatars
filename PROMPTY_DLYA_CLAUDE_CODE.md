# Промпты для Claude Code — по чатам

## Зачем делить на чаты

Весь проект в одной сессии Claude Code собирать нельзя: контекст переполнится, и качество поплывёт (Claude начнёт забывать ранние решения и ломать готовое). Правило: **один этап = один новый чат**. Между чатами всё держится на двух вещах — файле `TZ_AVATAR_BOT.md` и уже написанном коде в репозитории. Поэтому каждый новый чат начинается с того, что Claude Code читает ТЗ и существующий код.

## Подготовка (один раз)

1. Создай папку проекта, открой её в VS Code.
2. Положи туда `TZ_AVATAR_BOT.md` (в корень).
3. Запусти Claude Code в этой папке.
4. Инициализируй git (`git init`) — чтобы между чатами был «якорь» состояния.

## ЧАТ 1 — Каркас + Этап 1 (текст, персона, база знаний, чаты)

Вставь это первым сообщением:

```
Прочитай файл TZ_AVATAR_BOT.md целиком — это техническое задание проекта.
Подтверди, что понял общую картину, стек и правила из раздела 0
(особенно: сверяться с документацией API, не выдумывать эндпоинты,
без оверинжиниринга, секреты только в .env).

Затем реализуй ТОЛЬКО Этап 1 (раздел 2 ТЗ). Не трогай Этапы 2–4.

Порядок работы:
1. Сначала покажи план: структуру файлов и что в каждом будет. Жди моего «ок».
2. После «ок» — создавай проект: requirements.txt, config.py, .env.example,
   .gitignore, схему БД (db/schema.sql) и store.py (пока SQLite-режим),
   затем bot/persona_kb.py, bot/memory.py, bot/brain.py, bot/admin.py,
   bot/handlers.py, main.py.
3. Создай шаблон persona.md и положи 1 тестовый файл в knowledge/
   (например knowledge/test.md с парой фактов обо мне), чтобы я мог проверить КБ.
4. В конце дай инструкцию: как поставить зависимости и запустить локально,
   и чеклист проверки по критериям приёмки Этапа 1.

Важно: prompt caching на блоках persona+КБ обязателен — сверься с актуальной
документацией Anthropic по синтаксису cache_control перед тем как писать brain.py.
Модель: claude-sonnet-4-6.
```

Когда Этап 1 заработает: `git add -A && git commit -m "stage 1: text bot + KB + per-chat memory"`. Закрывай чат.

```

Сначала план, потом — после моего «ок» — реализация. В конце — инструкция проверки.

Заработало → коммит `stage 2: voice in/out`. Закрывай чат.

## ЧАТ 3 — Этап 3 (видео-кружочки, провайдеры, очередь)

Это самый большой этап. Новый чат:

```
Это проект Telegram-аватара. Прочитай TZ_AVATAR_BOT.md (раздел 4 — Этап 3)
и изучи существующий код. Этапы 1–2 готовы, не ломай их.

Реализуй Этап 3: видео-кружочки через переключаемых провайдеров Sora 2 и Veo.
Ключевое:
- video/base.py — интерфейс VideoProvider (см. ТЗ 4.1);
- video/veo.py — вход ФОТО из identity/ (несколько референсов, bind-subject);
- video/sora2.py — вход CAMEO по SORA_CAMEO_ID (Sora НЕ берёт фото лица!);
- video/postprocess.py — ffmpeg в кружок 640×640, ≤60 сек;
- bot/jobs.py — АСИНХРОННАЯ очередь в таблице video_jobs:
  диалог не блокируется, бот сразу отвечает текстом/голосом + "записываю кружочек",
  воркер генерит в фоне и присылает send_video_note;
- дневной лимит и кэш video_clips/.

Перед кодом КАЖДОГО провайдера сверься с актуальной документацией
(OpenAI Sora 2 Video API; Google Veo) — как именно вызывается генерация,
как поллить статус, как передаётся идентичность. Не выдумывай параметры.
Если у какого-то провайдера фичи нет — честно скажи и предложи fallback.

Сначала план (особенно как устроена очередь и интерфейс провайдера),
потом после «ок» — реализация. В конце — инструкция проверки и переключения
VIDEO_PROVIDER=veo|sora2.
```

Заработало → коммит `stage 3: video circles (sora2+veo) + async queue`. Закрывай чат.

Подготовь проект к деплою:

- Дай ГОТОВЫЙ db/schema.sql для Supabase (users, messages, video_jobs, daily_counters)

  создать buckets identity и videos.
- Подготовь деплой на Render как Background Worker (always-on):
  что вписать в build/start команды, как поставить ffmpeg, какие env задать.
  Дай пошаговую инструкцию словами — что мне нажимать в дашборде Render.
- main.py должен корректно работать в обоих режимах (local / prod).

Сначала план, потом после «ок» — реализация и инструкции.
```

Заработало → коммит `stage 4: deploy render+supabase`. Готово.

## Общие правила для всех чатов

- **Всегда начинай новый чат с «прочитай TZ + изучи существующий код»** — это переносит контекст между сессиями.
- Если внутри одного этапа чат стал очень длинным и Claude начал путаться — закоммить рабочее, открой новый чат, скажи «продолжаем этап X, вот что уже сделано, изучи код» и продолжи.
- Всегда требуй **сначала план, потом код** — так ловишь ошибки до их написания.
- После каждого этапа — **git commit**. Это твоя страховка и якорь контекста.
- Если Claude Code хочет добавить Docker/Redis/Celery/векторную базу — напомни про раздел 6 ТЗ «чего не делать».
- Тестируй каждый этап реально в Telegram перед переходом к следующему.
